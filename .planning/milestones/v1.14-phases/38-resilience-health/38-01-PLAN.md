# Phase 38 Plan 01: Resilience & Health Hardening

## Objective
Fast-fail outbound calls when session is unhealthy, verify recovery with status polling, and prevent unhandled rejections in queue drain.

## Context
- @src/http-client.ts — callWahaApi retry loop
- @src/health.ts — attemptRecovery, sessionHealthStates
- @src/inbound-queue.ts — drain() finally block

## Tasks

### Task 1: Circuit breaker in callWahaApi (RES-01)
type="auto"

Add optional `session` field to `CallWahaApiParams`. Add a module-level health checker callback (`setSessionHealthChecker`) that health.ts registers on startup. In `callWahaApi()`, before the retry loop, if session is provided and health checker reports "unhealthy", throw immediately with descriptive error.

**Done criteria:** callWahaApi fast-fails for unhealthy sessions without entering retry loop.

### Task 2: Recovery status polling (RES-02)
type="auto"

In `attemptRecovery()`, after restart POST returns 200, poll `GET /api/sessions/{session}` every 2s for up to 30s. Parse response for `status` field. Only mark `outcome="success"` when status is "CONNECTED" (case-insensitive). If 30s elapses without CONNECTED, mark `outcome="timeout"` with descriptive error.

**Done criteria:** Recovery only reports success when session is actually CONNECTED.

### Task 3: Wrap drain() finally block (CON-02)
type="auto"

In `InboundQueue.drain()` finally block, wrap both `onQueueChange?.(this.getStats())` and the recursive `this.drain()` call in their own try/catch blocks to prevent unhandled rejections from propagating.

**Done criteria:** Neither onQueueChange nor recursive drain can throw unhandled rejections.
