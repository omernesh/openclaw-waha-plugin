## 6. Tech Debt Summary

**Total: 11 items across 7 phases.**

See frontmatter `tech_debt` block for the full itemized list.

### Key items by severity

**High — functional gap (dead code path)**

1. **Phase 1: Dead configureReliability() config wiring.** The function is exported but never called at plugin startup. Config fields `timeoutMs`, `rateLimitCapacity`, and `rateLimitRefillRate` are never read from plugin config — hardcoded defaults are always used. Plan 01-03 SUMMARY incorrectly claimed this was deployed and verified.

**Medium — documentation drift**

2. **Phase 6: 14 stale requirement statuses.** RULES-01 through RULES-14 are fully implemented and verified in code, but REQUIREMENTS.md still shows them as "Planned". Any future reader of REQUIREMENTS.md will get a false picture of milestone completeness.

3. **Phases 8–11: 15 plan-internal requirements not in REQUIREMENTS.md.** UI-01–04, UX-01–04, DIR-01–04, DASH-01, SESS-01, LOG-01 exist only in phase plan files. They are invisible to milestone-level tracking.

**Medium — missing verification artifacts**

4. **Phase 7: No VERIFICATION.md, no VALIDATION.md.** Phase work was delivered (SQLite pagination, save-restart overlay, group filter timeout) but there is no formal verification record. This is the only phase in v1.10 with completely absent verification artifacts.

**Low — pending human verification**

5. **Phases 4, 9, 10, 11: 18+ items pending browser verification.** These are visual/behavioral items that cannot be verified by code inspection alone. They require a human to open the admin panel in a browser and confirm the behavior. The code is believed correct but is unconfirmed.

## 8. Recommendations

Ordered by impact:

1. **Update REQUIREMENTS.md**: Mark RULES-01 through RULES-14 as Complete. Add entries for UI-01–04, UX-01–04, DIR-01–04, DASH-01, SESS-01, LOG-01 with status Satisfied. This resolves 29 of the 11 tech debt items (the stale-status and missing-registration categories).

2. **Create Phase 7 VERIFICATION.md retroactively**: Document what was delivered (SQLite pagination, save-restart overlay, group filter timeout) and mark the verification method as retrospective code review. This closes the only phase with completely absent artifacts.

3. **Resolve configureReliability() dead code**: Either (a) call `configureReliability(pluginConfig)` in the plugin startup path so user config actually takes effect, or (b) remove the function and document that timeout/rate-limit values are hardcoded. Option (a) is the right fix; option (b) is honest but removes a capability.

4. **Complete human browser verification**: For Phases 4, 9, 10, and 11, open the admin panel in a browser and walk through the 18+ pending visual/behavioral items. Update the relevant VERIFICATION.md files with results.

5. **Accept Nyquist non-compliance for v1.10**: All phases were verified via code review. Formally note this acceptance in the milestone record. For v1.11, decide upfront whether to run the Nyquist wave protocol or continue with code-review-only verification.
