3. **`handleJoin`** (internal):
   - If `args` matches `chat.whatsapp.com/` OR is a 22+ char alphanumeric string → treat as invite link
     - Extract invite code: for URLs, take the path segment after `chat.whatsapp.com/`; for raw codes, use as-is
     - Call `joinWahaGroup({ cfg: config, inviteCode, accountId: account.accountId })`
     - On success: reply `Joined group ✓` via `sendWahaText({ cfg: config, to: chatId, text, accountId: account.accountId, bypassPolicy: true })`
     - On error: reply `⚠️ Could not join: {error.message}`
   - Else → treat as group name search
     - Call `resolveWahaTarget({ cfg: config, query: args, type: "group", accountId: account.accountId })`
     - If 1 match with confidence >= 0.8 → join directly (need to get invite code first via group JID — actually groups found by name are groups the bot session can see, so call `joinWahaGroup` won't work since bot is already in them. For /join by name, the use case is: user wants to join a group they're NOT in but know the name of. Since WAHA only joins via invite code, and we can't get an invite code for a group we're not in, the realistic flow is: search groups via resolveWahaTarget, if exact match found call joinWahaGroup. BUT joinWahaGroup requires an inviteCode, not a groupId. So for name-based join, we need to get the invite code from the matched group via getInviteCode first.)
     - CORRECTION: For /join by name — resolve the name to a group JID, then fetch the invite code via `callWahaApi` GET `/api/{session}/groups/{groupId}/invite-code`, then call `joinWahaGroup` with that code. If the bot is already in the group, reply "Already in {name}". If the group isn't found, reply "No groups matching '{args}' found".
     - Actually, re-reading the CONTEXT: "Join supports both invite links AND group names". The name-based join resolves to a group the user wants to join. The bot searches its known groups (via resolveWahaTarget) and if found, it means the bot is ALREADY in that group. So name-based /join should search ALL groups (including ones from directory/WAHA search). For simplicity and per CONTEXT decision: use resolveWahaTarget with type "group" which searches groups the bot session has access to.
     - If single high-confidence match (confidence >= 0.8): reply "Already a member of {name}" (since resolveWahaTarget only returns groups the bot is in)
     - If multiple matches (ambiguous): send numbered candidate list (top 5), store pending selection with type "join" via `getDirectoryDb(account.accountId).setPendingSelection(senderId, { type: "join", groups: candidates, durationStr: null })`. Store across ALL account DBs (same pattern as shutup.ts). Reply: "Multiple groups match '{args}':\n\n1) Group A\n2) Group B\n\nReply with a number to select."
     - If 0 matches: reply "No groups matching '{args}' found"

4. **`handleLeave`** (internal):
   - If `args` is empty: reply "Usage: /leave <group-or-channel-name>"
   - Call `resolveWahaTarget({ cfg: config, query: args, type: "auto", accountId: account.accountId })` — search both groups and channels
   - Filter matches to only groups (`@g.us`) and channels (`@newsletter`)
   - If single high-confidence match (>= 0.8):
     - If JID ends with `@newsletter` → call `unfollowWahaChannel({ cfg: config, channelId: jid, accountId: account.accountId })`
     - Else → call `leaveWahaGroup({ cfg: config, groupId: jid, accountId: account.accountId })`
     - Reply: `Left "{name}" ✓`
     - On error: reply `⚠️ Could not leave: {error.message}`
   - If multiple matches: send numbered list (top 5), store pending selection with type "leave". Reply: "Multiple matches for '{args}':\n\n1) Group A\n2) Channel B\n\nReply with a number to select."
   - If 0 matches: reply "No groups or channels matching '{args}' found"

5. **`handleList`** (internal):
   - Parse subcommand from args: detect "group" or "groups" → filter groups only; "channel" or "channels" → filter channels only; empty/other → show all
   - Call `getWahaGroups({ cfg: config, accountId: account.accountId })` — returns dict, use `Object.values()`
   - Call `getWahaChannels({ cfg: config, accountId: account.accountId })`
   - Format output with emoji prefixes:
     ```
     📱 Groups (N):
     1. Group Name
     2. Another Group

     📢 Channels (N):
     1. Channel Name
     ```
   - If filtered to groups only, show only the groups section. Same for channels.
   - If no results: reply "Not a member of any {groups/channels/groups or channels}"
   - Send via `sendWahaText` with `bypassPolicy: true`

6. **`handleCommandSelectionResponse`** — handles numbered replies for /join and /leave pending selections:
   ```typescript
   export async function handleCommandSelectionResponse(
     pending: PendingSelectionRecord,

   ): Promise<boolean>
   ```
   - Parse number from `text.trim()`
   - Validate 1-N range
   - If valid:
     - For type "join": the selected group is one the bot is already in, so reply "Already a member of {name}" (since resolveWahaTarget only finds groups bot has access to)
     - For type "leave": execute leave (leaveWahaGroup or unfollowWahaChannel based on JID), reply confirmation
   - If invalid: reply "Invalid selection. Reply with a number (1-{N})." and return false (don't clear pending)
   - Return true when handled (clear pending)
