    - "resolves names": each recipient string passed through autoResolveTarget before send
    - "caps at 10": throws error if recipients.length > 10
    - "no fail-fast": if recipient 2 of 3 fails, recipient 3 still gets attempted
    - "per-recipient results": returns array of {recipient, status: "sent"|"failed", error?} plus sent/failed counts
    - "requires text": throws if text param is empty
    - "requires recipients": throws if recipients array is empty
  </behavior>
  <action>
    1. In channel.ts, create async function handleSendMulti(cfg: CoreConfig, params: Record<string, unknown>):
       - Extract recipients via toArr(params.recipients) — handles both string and array
       - Extract text via String(params.text || "")
       - Extract optional replyToId via params.replyToId ? String(params.replyToId) : undefined
       - Validate: throw if !text, throw if recipients.length === 0, throw if recipients.length > 10
       - Loop sequentially over recipients:
         - try: resolve name via autoResolveTarget(cfg, recipient), then sendWahaText({ cfg, to: resolved, text, replyToId })
         - catch: push { recipient, status: "failed", error: err.message }, continue
         - success: push { recipient, status: "sent" }
       - Return { results, sent: count, failed: count }
    2. Add "sendMulti" to UTILITY_ACTIONS array
    3. Add to ACTION_HANDLERS: `sendMulti: handleSendMulti`

       - Mock autoResolveTarget and sendWahaText
       - Test all 7 behaviors listed above
       - Test that sends are sequential (not parallel) by checking call order

    DO NOT modify the existing "send" action — sendMulti is a NEW separate utility action.
    DO NOT parallelize sends — sequential loop respects rate limiter.
    DO NOT support media — text only per user decision (media multi-send deferred).
