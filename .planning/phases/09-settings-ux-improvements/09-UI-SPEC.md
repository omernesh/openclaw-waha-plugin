# Phase 9 — UI Design Contract

> Visual and interaction contract for Phase 9: Settings UX Improvements.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — custom inline CSS in monitor.ts template string |
| Icon library | none — unicode characters and text labels only |
| Font | system-ui, sans-serif (inherited from body rule at monitor.ts line 271) |

**Note:** This is a server-side HTML/JS admin panel embedded as a TypeScript template string in `src/monitor.ts`. There is no build pipeline, no component framework, and no external CSS file. All design tokens are CSS literals defined in the style block within the template string. Design system means: reuse the exact hex values and class patterns already established in that style block.

---

## Spacing Scale

All spacing is in multiples of 4px, matching existing admin panel conventions.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Icon-to-label gap in `.field label`, inline `.tip` badge margin |
| sm | 8px | Padding inside `.tip::after` tooltip popup (6px 10px established), clear button right offset |
| md | 16px | Contact settings panel padding, tooltip popup offset |
| lg | 24px | Search bar wrapper horizontal padding |
| xl | 32px | Not used in this phase |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Tooltip popup uses `padding: 6px 10px` (6px is off the 4px scale but is the established pattern — do not change it)
- Touch targets for the search clear button: minimum 28px height to match the existing `.dir-search` input rendered height
- `.tip` badge is 16px x 16px (existing and established — do not change)

## Typography

All sizes are expressed as rem, matching the existing CSS. Pixel equivalents assume 16px root.

| Role | Size | Weight | Line Height | Context |
|------|------|--------|-------------|---------|
| Body text | 0.88rem (14px) | 400 | 1.5 | General text, inputs, `.dir-search` |
| Label | 0.82–0.85rem (13–14px) | 400 | 1.4 | `.field label`, `.settings-field label` |
| Section heading | 0.85rem (14px) | 600 | 1.2 | `.card h2`, `.settings-section summary` — uppercase, letter-spacing 0.06em |
| Tooltip content | 0.75rem (12px) | 400 | 1.4 | `.tip::after` — width 220px, white-space normal |

Weights used: 400 (regular) and 600 (semibold) only. Do not introduce 500 or 700 in new elements.

Tooltip text (UX-02 specific): All `data-tip` attribute values inherit 0.75rem / line-height 1.4 from the `.tip::after` rule. Write tooltip text as plain prose. No markdown, no newlines. Avoid apostrophes — rephrase or use the HTML entity `&#39;` if unavoidable.

## Component Contracts

### UX-01: DM Policy Pairing Mode

Two outcomes after live test. Both require an updated tooltip on the DM Policy label.

If pairing works end-to-end:
- Keep the pairing option element as-is
- Updated `data-tip` for DM Policy: `"How to handle DMs from unknown senders. open: accept all. closed: block all. allowlist: only contacts in Allow From list. pairing: sender receives a numeric code; owner must approve via /pair command."`

If pairing does not work (code reply never delivered):
- Add `disabled` attribute to the pairing option element
- Change option display text to: `pairing (not available)`
- Updated `data-tip` for DM Policy: `"How to handle DMs from unknown senders. open: accept all. closed: block all. allowlist: only contacts in Allow From list. pairing: not supported in current SDK integration."`
- Add inline HTML comment: `UX-01: pairing disabled - SDK integration not verified`

### UX-02: Contact Settings Tooltips

Four fields in `buildContactCard()` each receive a `.tip` span appended inside their `<label>`. Use the identical HTML pattern as all existing Settings section tooltips (monitor.ts lines 499–619).

Pattern: `<label>FIELD_NAME <span class="tip" data-tip="TOOLTIP_TEXT">?</span></label>`

| Field | Tooltip copy (data-tip value) |
|-------|-------------------------------|
| Mode | `Active: bot responds to this contact. Listen Only: messages arrive but bot does not reply.` |
| Mention Only | `When checked, bot only responds if it is explicitly @mentioned in the message.` |
| Custom Keywords | `Comma-separated regex patterns. Bot responds only if the message matches one. Overrides global keyword filter for this contact.` |
| Can Initiate | `When checked, bot is allowed to send the first message to this contact. Uncheck to prevent unsolicited outbound messages.` |

The `?` character is the only visual indicator for the tooltip trigger. No icons, no SVGs, no emoji.

### UX-03: Group Filter Tag Input and Trigger Operator

Tag input replacement:
- Replace the plain text input element `id="gfo-patterns-{sfx}"` with a container div `id="gfo-patterns-cp-{sfx}"`
- Call `createTagInput('gfo-patterns-cp-' + sfx, { placeholder: 'hello, help, bot' })` strictly AFTER the HTML string has been assigned to the panel element (not during string construction)
- Store the returned instance in a `gfoTagInputs` object keyed by `sfx` (sanitized JID): create if not exists pattern
- Update `saveGroupFilter` to call `.getValue()` on `gfoTagInputs[sfx]` instead of reading from a DOM input element

Trigger operator select element:
- Element: `<select id="gfo-operator-{sfx}">` with two options: `OR` (value `"OR"`, label `"OR – match any keyword"`) and `AND` (value `"AND"`, label `"AND – match all keywords"`)
- Default: `OR`
- Label: `Trigger Operator` with `.tip` span: `data-tip="OR: message matches if it contains any keyword. AND: message must contain all keywords."`
- Position: directly below the keyword filter enabled checkbox, above the keywords tag input
- Visual style: matches existing `.field select` pattern — `background: #0f172a; border: 1px solid #334155; color: #e2e8f0; border-radius: 5px; padding: 6px 10px; font-size: 0.85rem; width: 100%`
- Save/load field name: `triggerOperator` in the PUT `/api/admin/directory/:jid/filter` request body

### UX-04: Tab Switch, Search Clear, Label Rename

Tab label rename:
- Change the Newsletters tab button visible text from `Newsletters` to `Channels`
- The JS identifier `'newsletters'` and the API query parameter `&type=newsletter` stay unchanged
- Only the visible label text changes

Tab switch clears search:
- In `switchDirTab()`, insert two lines before `dirOffset = 0`: get element `dir-search` and set its `.value` to empty string (guard with null check)

Search bar clear button:
- Wrap the existing `dir-search` input in a position-relative container div with `flex:1` (the wrapper carries flex growth, remove it from the input)
- Add a button `id="dir-search-clear"` positioned absolute, right `8px`, vertically centered via `top:50%; transform:translateY(-50%)`
- Button content: Unicode multiplication sign `&#x2715;` (renders as ×)
- Button base style: `background:none; border:none; color:#94a3b8; cursor:pointer; font-size:1rem; line-height:1; padding:0 4px`
- Hover color: `#e2e8f0` (apply via `onmouseover`/`onmouseout` inline attributes, matching the project's vanilla JS pattern)
- Click handler: named function `clearDirSearch()` that sets `dir-search` input value to empty string, resets `dirOffset` to 0, and calls `loadDirectory()`
- The button is always visible (not conditionally shown based on input content)

## Interaction Contracts

### Tooltip
- Trigger: CSS hover only — no JavaScript
- Popup appears above the `?` badge (`bottom: calc(100% + 6px)`)
- Popup width: 220px fixed
- Transition: opacity 0.15s
- No click or keyboard interaction required

### Tag Input (reused from Phase 8)
- Add tag: press Enter, comma, or space after typing a value
- Remove tag: click the × on the tag bubble
- Tag visual: `background: #1e3a5f; color: #7dd3fc; border-radius: 4px; font-size: 0.75rem; padding: 2px 8px` (matches existing `.tag` class)
- Placeholder shown when input is empty

### Trigger Operator Select
- Interaction: standard HTML select — no custom dropdown
- Change is applied only on Save click (same pattern as all other group filter fields)

### Search Clear Button
- Single click triggers `clearDirSearch()`
- Always visible, no conditional display
- No loading state

### Tab Switch
- After this phase: switching tabs additionally clears the search input value before loading directory
