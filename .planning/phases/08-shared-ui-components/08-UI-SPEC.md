# Phase 8 — UI Design Contract

> Visual and interaction contract for shared UI components used across Settings, Dashboard, and Directory sections. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (vanilla HTML/CSS/JS embedded in monitor.ts) |
| Icon library | none (CSS-only indicators, Unicode glyphs for arrows) |
| Font | system-ui, sans-serif (inherited from body) |

**Context:** The admin panel is a single-page web UI served as an embedded HTML string from `src/monitor.ts`. There is no React, no Tailwind, no build step. All components are written as vanilla HTML/CSS/JS string literals. The design system is a hand-crafted dark theme consistent across ~2,900 lines of monitor.ts. Phase 8 components must integrate seamlessly into this existing system — they are not standalone — they will be injected as reusable HTML snippets or JS factory functions within monitor.ts.

## Component Inventory

Phase 8 introduces 4 reusable components. Each is implemented as a JS factory function or self-contained HTML/CSS block within `monitor.ts`. All components render in the existing dark theme.

### Component 1: Name Resolver (UI-01)

**Purpose:** Replace raw JID/LID/phone strings with human-readable contact names everywhere they appear in the panel.

**Behavior:**
- Calls `GET /api/admin/directory/:jid` to resolve a JID to a display name
- Shows resolved name in `text-primary` (`#e2e8f0`)
- Shows raw JID as subtitle in `text-muted` (`#64748b`) in monospace font, font-size 12px
- Loading state: skeleton shimmer (animated `background: linear-gradient(90deg, #1e293b 25%, #334155 50%, #1e293b 75%)`)
- Failed resolution: show raw JID only in `text-muted`
- Resolution is best-effort — never blocks rendering

**Visual anatomy:**
```
[Avatar 40×40] [Display Name         ]
               [raw-jid@c.us    mono  ]
```

**Avatar:** colored circle with first letter of display name (or "?" if unresolved). Avatar background colors rotate through: `#1e3a5f`, `#1a3a2a`, `#3a1a2a`, `#2a2a1a` (4-color set, chosen by hash of JID).

### Component 2: Tag Input (UI-02)

**Purpose:** Replace plain textarea fields for JID lists (God Mode Users, Allow From, Group Allow From) with tag-style bubble inputs.

**Behavior:**
- Accepts comma, space, or Enter to confirm a tag
- Each confirmed tag renders as a pill bubble
- Pill: `background: #0ea5e9`, `color: #fff`, `font-size: 0.8rem`, `padding: 3px 10px`, `border-radius: 9999px`, monospace font
- Each pill has an `×` remove button: inline, `color: rgba(255,255,255,0.7)`, `cursor: pointer`, 16px click zone
- Hovering `×` changes opacity to 1.0
- Pill container is a flex-wrap div styled as an input field: `background: #0f172a`, `border: 1px solid #334155`, `border-radius: 6px`, `padding: 8px`
- When focused (cursor in text input area): `border-color: #38bdf8`
- Input placeholder: "Type JID or phone, press Enter"
- Backspace on empty text input removes last tag
- Tab key completes current typed text as a tag (if non-empty)
- getValue() returns array of trimmed strings
- setValue(arr) populates tags from array
- Renders consistently with existing `.field input` styling

**States:**
- Empty: shows placeholder text in `text-muted`
- Has tags + focused: blue border
- Has tags + blurred: default border
- Invalid tag (empty string after trim): silently ignored — no error state

### Component 3: Contact Picker (UI-03)

**Purpose:** Fuzzy-search contact selector with multi-select. Used for God Mode Users and allowlist fields to replace raw JID textarea with a search-and-pick UI.

**Behavior:**
- Search input calls `GET /api/admin/directory?search={query}&limit=20` on keydown with 300ms debounce
- Results list appears as dropdown below input, `max-height: 240px`, `overflow-y: auto`
- Each result row: Avatar (32×32) + Display Name + JID subtitle
- UTF-8 aware: Hebrew and English names must render correctly (no font override needed — system-ui handles both)
- Selected items appear as name chips above the input
- Multi-select: clicking a result adds to selection (does not close picker)
- Selected items are visually marked with a checkmark prefix in the dropdown (Unicode `✓` in `#10b981`)
- Clicking a selected item in the dropdown removes it from selection
- Dropdown closes on outside click or Escape key
- "No results" state: shows "No contacts found" in `text-muted`, centered, padding 16px
- Loading state: shows single row with shimmer animation

**Name chip (selected item):**
- `background: #1e3a5f`, `color: #7dd3fc`, `font-size: 0.75rem`, `padding: 4px 12px`, `border-radius: 4px`
- Remove `×` button on right side, `color: #7dd3fc`, 16px click zone
- Matches existing `.tag` style

**Dropdown container:**
- `background: #1e293b`, `border: 1px solid #334155`, `border-radius: 8px`
- `box-shadow: 0 4px 16px rgba(0,0,0,0.4)`
- `position: absolute`, `width: 100%`, `z-index: 300`

**Scroll behavior:** Dropdown scrollbar uses default browser styling — no custom scrollbar.

**Paired JID handling (UI-04 requirement):**
When a contact is selected for God Mode Users, the component automatically resolves and includes both `@c.us` and `@lid` JIDs as a paired chip group. The chip label shows the display name; hovering shows both JIDs in a tooltip (using existing `.tip` pattern). Removing the chip removes both JIDs.

### Component 4: God Mode Users Field (UI-04)

**Purpose:** Replaces the raw textarea for `godModeSuperUsers` with the Contact Picker (Component 3) configured for paired JID mode.

**Behavior:** Wraps Component 3 with paired JID resolution. On save, serializes selected contacts to `[{ identifier: "jid@c.us" }, { identifier: "lid@lid" }]` array format that the config API expects.

**Visual:** Inherits all Contact Picker styling. No additional visual chrome.

## Implementation Notes for Executor

1. **No build step.** All component code goes inside the HTML string template in `monitor.ts` as `<style>` additions and JS factory functions.

2. **Style additions** go in the existing `<style>` block (line ~269 of monitor.ts). Add new class names with `.component-name-` prefix to avoid collision.

3. **JS factory functions** go in the `<script>` block near existing utility functions. Pattern: `function createTagInput(containerId, opts) { ... }`.

4. **Initialization** replaces existing textarea field construction at the point where those fields are rendered.

5. **Existing textarea values** must migrate cleanly: `setValue()` on the new component is called with the same data that previously set the textarea value.

6. **Save path unchanged:** The `buildPayload()` / `collectConfig()` JS function that reads form fields must call `getValue()` on new components instead of reading textarea values.

7. **No fetch calls from components directly** — components receive data via constructor arguments or call existing API functions already in the script block.

8. **Accessibility minimum:** All interactive elements must have `aria-label`. Focus must be trapped in picker dropdown while open (Tab cycles through results, Escape closes).
