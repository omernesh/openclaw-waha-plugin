2. Add API routes following the existing pattern (search for how other /api/admin/ routes are registered):

   **GET /api/admin/modules** — List all registered modules
   - Returns: `{ modules: [{ id, name, description, version, enabled, assignmentCount }] }`
   - Gets list from `getModuleRegistry().listModules()`
   - For each module, query DirectoryDb for assignment count

   **PUT /api/admin/modules/:id/enable** — Enable a module
   - Calls `getModuleRegistry().enableModule(id)`
   - Returns: `{ ok: true }`

   **PUT /api/admin/modules/:id/disable** — Disable a module
   - Calls `getModuleRegistry().disableModule(id)`
   - Returns: `{ ok: true }`

   **GET /api/admin/modules/:id/assignments** — List chat assignments for a module
   - Returns: `{ assignments: string[] }` (JID list)
   - Queries DirectoryDb.getModuleAssignments(id)

   **POST /api/admin/modules/:id/assignments** — Assign chat to module
   - Body: `{ jid: string }`
   - Calls DirectoryDb.assignModule(id, jid)
   - Returns: `{ ok: true }`

   **DELETE /api/admin/modules/:id/assignments/:jid** — Unassign chat from module
   - Calls DirectoryDb.unassignModule(id, jid)
   - Returns: `{ ok: true }`

3. All routes use the same accountId resolution pattern as existing admin routes.
